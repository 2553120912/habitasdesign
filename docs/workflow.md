# 执行步骤 / 协作流程

## 核心原则

> 小步快走，步步确认。不一口气做太多。

## 开发流程

```
1. 讨论 ──→ 2. 规划 ──→ 3. 实施 ──→ 4. 检查
                                         ↑
                                   用户反馈后可回到 1
```

### 1. 讨论
- 用户提出想法或需求
- Claude 询问细节、澄清模糊点

### 2. 规划
- Claude 进入计划模式（Plan Mode）
- 写出具体改动方案
- **用户确认后**才进入实施

### 3. 实施
- 每次改动范围控制在 1-3 个文件
- 改完立刻打开浏览器让用户检查
- 不做"顺便改一下"的超范围修改

### 4. 检查
- 浏览器中实际查看效果
- 更新开发日志
- 确认无误后进入下一轮讨论

## 开发日志规范

每次开发结束后，在 `devlog/` 下创建或更新当日日志：

```markdown
# YYYY-MM-DD 开发日志

## 今日完成
- [xxx] 具体事项

## 待办事项
- [ ] 待处理事项

## 备注
其他说明
```

## Git 提交规范（如启用）

- 使用中文提交信息
- 格式：`[类型] 简要描述`
- 类型：新增 / 修改 / 修复 / 优化 / 文档

## 检查清单

每次改动前确认：
- [ ] 已阅读相关 docs/ 规范文件
- [ ] 改动范围明确、可控
- [ ] 不会破坏现有功能

每次改动后确认：
- [ ] 浏览器中实际查看
- [ ] 手机端布局正常
- [ ] 更新了开发日志
